# Modelling review & the v2 model

How the Bookings Desk turns form into booking/foul probabilities, what was
wrong with the first model, and what the v2 model (this repo) does about it.

## The original model (v1)

Three layers:

1. **Risk score** — `risk = yc_per90 × 2 + fouls_per90`, a hand-weighted composite.
2. **Season P(card)** — a logistic `p = 1/(1+e^-(a + b·risk))` with slope `b`
   **fixed at 1.1** and only the intercept `a` fitted (anchored so the
   minutes-weighted average player hits the league base rate).
3. **Fixture P(card)** — `season_p × refFactor × derby` (multiplicative).

Fine as a **ranking screen**, but a **miscalibrated forecaster**. The problems,
most impactful first:

| # | Problem | Why it hurts |
|---|---------|--------------|
| a | Referee applied by multiplying the **probability** | `0.72 × 1.3 = 0.94`. On the odds scale it's `0.77`. Overstates by up to ~17pp exactly on the headline picks. |
| b | No **shrinkage** for sample size | 1 yellow in 500 mins reads as 0.18/90. 58 "qualified" players have 450–900 mins where rates are noise. |
| c | Weights/slope **guessed, not fitted** | `×2` and `b=1.1` are hand-picked; only the intercept sees data, so the spread is unvalidated. |
| d | Season-average **composite, not a match model** | A real forecast is event-level: P(≥1 card *this* match) given minutes, opponent, ref, venue. |
| e | No **exposure/minutes** term | A per-90 rate assumes 90 minutes; a 60-minute sub is lower. |
| f | No **opponent** effect on the player number | Opponent fouls-drawn/duel intensity matters (needs the `fw` field). |
| g | No **position** baseline | DMs/full-backs foul & book far more than forwards/keepers. |
| h | **Fouls not forecast** | They're only an input to risk, never an output (over/under markets). |
| i | No **out-of-sample validation** | Calibration was in-sample by construction. |

## The v2 model (implemented)

All the maths is pure and unit-tested in `assets/core.js` (`PLDCore`); the
parameters live in `data/model.js`, generated by `scripts/build-model.mjs`.

### Tier 1 — no new data (live now)

- **Empirical-Bayes shrinkage** (`PLDCore.shrinkRate`): pull each per-90 rate
  toward its **position mean**, weighted by exposure in matches —
  `rate = (events + mean·k) / (matches + k)`, `k = 6` matches. Fixes (b), (g).
- **Log-odds referee/derby** (`PLDCore.contextProb` / `scaleOdds`): multiply the
  **odds**, not the probability. Fixes (a).
- **Calibration metrics** (`PLDCore.brier`, `logLoss`, `reliability`): the Model
  Track Record now reports Brier and log-loss over scored picks vs a base-rate
  baseline. Fixes (i) (as data accrues).

### Tier 2 — fitted logistic GLM

- `PLDCore.glmProb(feats, coef)` scores a logistic with features
  `{yc90, foul90, DF, MF, FW}`. Coefficients live in `data/model.js`.
- **Basis `"season-prior"`** ships today: the prior reproduces the v1 logistic
  but on **shrunk** inputs and re-anchored — an honest, improvable starting
  point, *not* an empirical fit.
- **Basis `"match-fit"`**: run the real fit once match-level data exists —
  ```
  python3 data/harvest_history.py                       # FPL element-summary
  node scripts/build-model.mjs --fit data/match_history.json
  node scripts/backtest.mjs                             # prove it helps
  ```
  The fitter is IRLS (Newton-Raphson); the backtest walks forward gameweek by
  gameweek and reports out-of-sample Brier/log-loss for base vs prior vs a
  weekly refit. Fixes (c), (d), (g); (e)/(f) come in as features are added.

### Tier 3 — fouls forecast + two-stage card

- **Expected fouls** = `foulRate90 × expMinutes/90` (`PLDCore.expectedFouls`).
- **Over/under** via a **Negative-Binomial** tail (`PLDCore.nbTailProb`,
  overdispersed Poisson) → P(fouls over 1.5 / 2.5). Fixes (h).
- **Two-stage card** (`PLDCore.cardProbFromFouls`): mechanistic —
  `P(card) = 1 − exp(−E[fouls] × hazard)`, where `hazard` is the league
  cards-per-foul (anchored to the base rate) scaled by the referee. Surfaced in
  the player profile as a cross-check against the logistic.

### Recency-weighted fit (recent form counts more)

The match-level GLM fit weights each match row by **`0.97^(gameweeks in the
past)`** (`PLDCore.recencyWeight`, `data/model.js` `recencyDecay`), so recent
form counts for more than early-season noise — mirroring the match-model
recency decay on gameweekedge.co.uk. It applies in `scripts/build-model.mjs`
(the shipped fit) and in `scripts/backtest.mjs`'s weekly refit, so the
walk-forward score reflects the same weighting the live model uses. With no
gameweek on a row the weight is uniform (a no-op), so the season-prior basis
is unchanged.

## The calibration loop (server-verified accuracy)

The **Model Track Record** card self-scores in each visitor's browser. On top
of that, a server-side loop grades the model **in the open**, aggregated across
everyone — parity with Gameweek Edge's P5 prediction log:

- **`supabase/plb_predictions.sql`** — a service-role-only table (RLS on, no
  policies): one row per `(season, gw, element)` with the forecast season
  `pcard` and, once the gameweek finishes, `carded` (1/0).
- **`netlify/functions/log-predictions.js`** — a scheduled (`@hourly`) function.
  Before each deadline it logs the upcoming gameweek's season P(card) for every
  player with a fixture; once a gameweek finishes it backfills who was actually
  booked from the FPL result feed (`event/{gw}/live/`). It **reuses the shipped
  pure maths** (`assets/core.js` + `data/model.js` + `data/pl_data.js`, bundled
  via `included_files`) rather than re-implementing the model, so what is graded
  is exactly what the app shows — the referee-independent season P(card) (the
  per-fixture referee scaling is a client-local assignment and can't be graded
  server-side).
- **`netlify/functions/model-calibration.js`** — a public read returning only
  **aggregates**: sample size, observed booking rate, Brier vs a base-rate
  baseline, log-loss, a reliability curve, and the top-20-per-gameweek hit rate.
  The Guide shows it as **Live prediction accuracy** once the loop has graded a
  gameweek.

Both talk to Supabase over PostgREST with the service-role key (the same
dependency-free pattern as `insights.js`). The feature is optional: with no
`SUPABASE_SERVICE_ROLE_KEY` the logger no-ops and the card stays hidden.

## Honesty boundary

Tier 1 and Tier 3 are **fully data-grounded** from the shipped season data. The
Tier-2 **coefficients** ship as a **season-derived prior** (`basis:"season-prior"`)
because an empirical match-level fit needs history that the FPL API provides but
that isn't harvestable from every environment. The full fitting + backtest
machinery is implemented and unit-tested; running the harvest flips the model to
`"match-fit"` and the app shows the basis on the Model Track Record card.
